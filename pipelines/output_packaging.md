# pipelines/output_packaging.md

## Purpose

The output packaging stage prepares final artifacts after execution support and validation are complete. Its job is to organize deliverables into a clean, submission-ready package without inflating, faking, or overstating what was actually produced.

Packaging is the last step before handoff. It does not generate new work — it organizes existing verified work.

---

## What Packaging Includes

- **Organizing final files** — Group deliverables into a clear structure that matches the assignment scope
- **Keeping naming clean** — File names should be readable, consistent, and aligned with assignment requirements
- **Separating draft artifacts from final artifacts** — Drafts, working notes, and intermediate files must not be mixed into the final package
- **Including proof artifacts when they actually exist** — Screenshots, test outputs, and verification evidence are only packaged if they were truly captured
- **Preserving the distinction between generated work and verified work** — Generated content and manually verified content remain identifiable, never blended to imply more verification than occurred

---

## Common Output Types

| Output Type | Description |
|---|---|
| Final DOCX | Word document for written assignments, reports, or papers |
| Final PDF | Exported PDF for submission or distribution |
| Code files | Source files for programming assignments |
| Screenshot appendix | Captured proof of software execution or task completion |
| Slide deck | Presentation file (.pptx or equivalent) |
| Spreadsheet | Excel or equivalent file for data analysis assignments |
| QA or validation note | Summary of what was validated, what was generated, and what manual steps are still required |

---

## Packaging Checks

- **File names match assignment requirements** — Required naming conventions, course codes, or student identifiers are applied
- **Required sections are present** — All sections listed in the rubric or instructions are included in the final deliverable
- **Placeholders are removed** — No `[Your Name]`, `[Date]`, `TBD`, or unfilled fields remain
- **Screenshots or proof artifacts are only included if they actually exist** — No fabricated, simulated, or placeholder evidence
- **Final package is organized for handoff or submission** — Folder structure is clean, files are easy to locate, and nothing extraneous is included

---

## Rule

Packaging happens after execution support and validation. The system must not package missing proof as if it exists.
